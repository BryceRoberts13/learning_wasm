# Pong WASM Design Document

**Date:** 2025-03-12  
**Project:** learning_wasm / pong_v0

---

## Goal

Learn WebAssembly and Rust by building a minimal Pong game. MVP first, iterate later.

---

## Approach

**wasm-pack + minimal JavaScript** — Rust handles game logic; JavaScript handles canvas, input, and game loop. Standard pattern, good for learning.

---

## Architecture

1. **HTML** — Page with `<canvas>`, script loads WASM module
2. **JavaScript** — Game loop (`requestAnimationFrame`), keyboard input, calls into WASM each frame
3. **Rust/WASM** — Game state and logic: ball position/velocity, paddle positions, collision detection. Exports: `init()`, `tick(dt)`, `get_state()`, `set_paddle(player, direction)`
4. **JavaScript** — Draws current state to canvas each frame using data from WASM

## MVP Scope

- Two paddles, one ball
- Ball bounces off paddles and top/bottom walls
- Player 1: W/S. Player 2: Arrow keys
- No score or win condition yet

---

## Data Flow

1. JS loads WASM → calls `init()` → gets initial state
2. Each frame: JS calls `tick(delta_ms)` → then `get_state()` → draws to canvas
3. On keydown/keyup: JS calls `set_paddle(player, direction)` (-1, 0, or 1)

---

## Error Handling (MVP)

- If WASM fails to load, show error in console or on page
- No extra validation; assume valid inputs

---

## Tech Stack

- **Rust** — wasm-bindgen, wasm-pack
- **JavaScript** — Vanilla JS, Canvas API
- **Build** — wasm-pack build, static file serving (e.g. `python -m http.server` or similar)
